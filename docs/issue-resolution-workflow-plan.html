Agent-Ready Issue Resolution Workflow

A sequential, dependency-aware workflow that implements each eligible GitHub issue, reviews and repairs its pull request until clean, merges it, and returns the repository to an updated main before selecting the next issue.

Plan status: draft only. The workflow has not been run. Deterministic Git and GitHub transitions described below must be implemented and validated before the first execution.

Workflow graph

    OPEN_PR --> REVIEW[Review agent: thermo-nuclear review plus code-review]
    REVIEW --> FINDINGS{Actionable findings?}
    FINDINGS -- Yes --> FIX[Review agent fixes, tests, commits, and pushes]
    FIX --> REVIEW
    FINDINGS -- No --> VERIFY[Run complete repository verification]
    VERIFY --> VERIFY_OK{Code-caused failures?}
    VERIFY_OK -- Yes --> FIX
    VERIFY_OK -- No --> MERGE[Deterministic transition: merge approved PR]

    SYNC --> CLOSED{Issue closed?}
    CLOSED -- No --> CLOSE[Close issue with PR and verification summary]
    CLOSED -- Yes --> FRONTIER
    CLOSE --> FRONTIER

Separation of responsibilities

Workflow control plane

Owns ordering, stop conditions, structured handoffs, and the outer issue loop.

  • Select only eligible issues with no open native blockers.
  • Process one issue at a time.
  • Do not advance after a failed implementation or review.
  • Re-query the dependency frontier after every merge.

Deterministic transition helper

Owns repeatable Git and GitHub state transitions rather than delegating decisions to prose prompts.

  • Create agent/issue-<number> from updated main.
  • Push and open exactly one PR targeting main.
  • Merge only after the review gate reports clean.
  • Restore updated main and assert a clean tree.

Implementation agent

Owns behavior changes through the TDD loop, not repository lifecycle decisions.

  • Read the issue title, body, labels, and comments.
  • Use confirmed public test seams.
  • Work red → green in vertical slices.
  • Run relevant checks and report exact results.

Review agent

Owns the review/fix loop on an already-created PR, not PR creation or merging.

  • Review standards and specification separately.
  • Fix every actionable finding and push the fix.
  • Repeat a fresh review until no findings remain.
  • Run the complete required verification suite.

Prompt boundaries

Prompt Contents Explicitly excluded
Shared implementation $tdd ${issue_url} Branch creation, PR creation, review, merge, and return to main.
Shared review $thermo-nuclear-code-quality-review $code-review PR creation, merge decisions outside the clean-review gate, and repository reset.
Per issue Optional common, implementation, and review additions keyed by issue number. Instructions that weaken repository rules, safety gates, tests, or verification.

Issue selection and dependency policy

  1. Read open issues through gh, excluding pull requests.
  2. Use issues labelled ready for agent, case-insensitively. If no open issue has that label, treat all open issues as eligible under the repository's current policy.
  3. Read GitHub's native issue_dependencies_summary.blocked_by value; do not infer dependencies from issue numbers.
  4. Choose only an issue with zero open blockers.
  5. Prefer an issue that blocks another eligible issue, then use the lowest issue number as the deterministic tie-breaker.
  6. Recompute the frontier after each completed issue because merging and closing can unlock the next issue.

Safety and completion gates

Before implementation

  • No unrelated tracked or untracked work.
  • main fast-forwards successfully.
  • The issue is open, eligible, unblocked, and claimed.
  • TDD seams are already established; otherwise stop for user confirmation.

Before merge

  • A fresh review reports zero actionable findings.
  • The PR contains only the selected issue's changes.
  • The feature branch and working tree are clean.
  • Required checks pass, or an external environment limitation is precisely documented.

Before next issue

  • The PR merge is confirmed.
  • The issue is confirmed closed.
  • Local main matches the updated remote via fast-forward pull.
  • The working tree is clean.

Required verification

The final clean review must run and report the repository's complete verification sequence:

  1. uv run ty check
  2. Pure Python unit tests
  3. Blender background smoke test using BLENDER_BIN
  4. Blender extension validation
  5. Installation ZIP build

A feature must not be reported as working unless tested. Interactive Blender behavior that remains untested must be identified explicitly.

Failure behavior and bounded execution

  • Stop immediately on dirty-state conflicts, failed state transitions, unresolved code failures, malformed agent output, or a review that cannot reach clean.
  • Never merge or close an issue with known actionable findings.
  • Return completed issue numbers, PR URLs, commits, review rounds, findings fixed, and exact verification results.
  • The workflow engine allows at most 32 agent calls. The design should keep deterministic transitions outside additional model calls so the current ten-issue batch fits safely.
  • If a per-run issue limit is reached, return batch-limit-reached; a later run resumes naturally because merged issues are closed.
